### [2026-05-10] Bug fixes ‚Äî state transition reload, timer/parts state guards, multi-tab auth

**Bugs found during testing:**

2. In `oficina/Detalhe.vue`, the timer could be started and stopped while the OS was in AGUARDA_APROVACAO. Stopping the timer in this state appeared to "reset" the counter because the session was shorter than 1 minute (and the backend accumulates whole minutes only). The parts search was also visible in AGUARDA_APROVACAO, which made no sense since the mechanic is waiting for client approval.
3. Using two browser tabs logged in as different users (mechanic and receptionist) caused the tabs to share auth state via localStorage. Refreshing the mechanic tab would load the receptionist's token, effectively logging out the mechanic.

**Prompt:**
There are three bugs. First, when you submit a state change the page sometimes shows an error even though it actually went through ‚Äî and if you try again it says it's already done. The page just wasn't refreshing properly after the transition. Second, the timer and parts panel were showing up when the OS is waiting for client approval, which makes no sense. They should only appear when the mechanic is actually working. Third, if you have two tabs open with different users logged in, they share the same session, so logging in on one overwrites the other. Fix all three.

**What was done:**
- `oficina/Detalhe.vue`: moved `await load()` to `finally` block in `confirmTransition`; added `ESTADOS_TRABALHO = ['EM_DIAGNOSTICO', 'EM_REPARACAO', 'AGUARDA_PECAS']`; added `canTimer` computed; updated `canAddParts` to use same whitelist; timer button now uses `:disabled="!canTimer"`; timer label shows "Indispon√≠vel neste estado" when blocked
- `ordens-servico/Detail.vue`: moved `await loadOS()` to `finally` block in `confirmTransition` ‚Äî same root cause, same fix
- `store/auth.js`: replaced all `localStorage` calls with `sessionStorage`; each tab now holds an independent token so testing with multiple accounts in separate tabs works correctly; trade-off: session does not survive the tab being closed (acceptable for an internal-use app)

### [2026-05-16] Fix ‚Äî `numero` column NOT NULL constraint crash on transfer creation

**Problem:**
Every attempt to create a transfer request returned a 500 Internal Server Error. The POST reached the backend and the INSERT was attempted, but MySQL rejected it immediately with:

```
IntegrityError: (1048, "Column 'numero' cannot be null")
```

**Root cause ‚Äî chicken-and-egg ordering:**
The `numero` field is generated as `TRF-{year}-{id:04d}`, where `id` is the auto-increment primary key assigned by MySQL. To know the `id`, you need to INSERT the row first. But the column was defined as `NOT NULL`, so MySQL rejected the INSERT because `numero` was still `None` at that point.

The repository's `create()` method was:
```python
pt = PedidoTransferencia(**kwargs)   # numero = None
self.db.add(pt)
self.db.flush()                       # ‚Üê MySQL rejects: numero cannot be null
pt.numero = f"TRF-{year}-{pt.id:04d}"
```

**First solution (hotfix):**
Set a UUID-based placeholder before the flush to satisfy the constraint, then overwrite it with the real number after getting the id:

```python
pt.numero = f"TEMP-{uuid.uuid4().hex}"  # satisfies NOT NULL + UNIQUE
self.db.add(pt)
self.db.flush()                           # gets id
pt.numero = f"TRF-{year}-{pt.id:04d}"   # overwrites with real numero
```

**Why it was rejected:**
Although functional, this is a workaround that masks the actual design flaw. The column is semantically nullable during the brief window between INSERT and the UPDATE ‚Äî the NOT NULL constraint was incorrectly applied. UUID strings as placeholder values are semantically meaningless and leave the door open to corruption if something fails between the two writes.

**Final solution (proper fix):**
Made `numero` nullable at both the ORM and DB level, which accurately reflects that the value genuinely does not exist until the row has an id:

- `models/transferencia.py`: changed `Mapped[str]` ‚Üí `Mapped[Optional[str]]` with `nullable=True`
- `repositories/transferencia_repository.py`: removed UUID placeholder; clean two-step flow (flush ‚Üí set numero)
- `alembic/versions/a1c3e5f7b9d2_...py`: migration to `ALTER TABLE pedidos_transferencia MODIFY COLUMN numero VARCHAR(50) NULL`

The column is still `unique=True`, so no two rows can ever share the same generated number. The `NULL` state is only possible inside the same uncommitted transaction.
