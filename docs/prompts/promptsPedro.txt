### [2026-05-10] Bug fixes вҖ” state transition reload, timer/parts state guards, multi-tab auth

**Bugs found during testing:**
1. After a state transition (both in `oficina/Detalhe.vue` and `ordens-servico/Detail.vue`), the UI did not update immediately. The mechanic/receptionist saw an error, tried to resubmit, and got "transition not permitted" because the first attempt had already succeeded. The status only refreshed when another action (like stopping the timer) triggered a reload.
2. In `oficina/Detalhe.vue`, the timer could be started and stopped while the OS was in AGUARDA_APROVACAO. Stopping the timer in this state appeared to "reset" the counter because the session was shorter than 1 minute (and the backend accumulates whole minutes only). The parts search was also visible in AGUARDA_APROVACAO, which made no sense since the mechanic is waiting for client approval.
3. Using two browser tabs logged in as different users (mechanic and receptionist) caused the tabs to share auth state via localStorage. Refreshing the mechanic tab would load the receptionist's token, effectively logging out the mechanic.

**Prompt:**
There are three bugs. First, when you submit a state change the page sometimes shows an error even though it actually went through вҖ” and if you try again it says it's already done. The page just wasn't refreshing properly after the transition. Second, the timer and parts panel were showing up when the OS is waiting for client approval, which makes no sense. They should only appear when the mechanic is actually working. Third, if you have two tabs open with different users logged in, they share the same session, so logging in on one overwrites the other. Fix all three.

**What was done:**
- `oficina/Detalhe.vue`: moved `await load()` to `finally` block in `confirmTransition`; added `ESTADOS_TRABALHO = ['EM_DIAGNOSTICO', 'EM_REPARACAO', 'AGUARDA_PECAS']`; added `canTimer` computed; updated `canAddParts` to use same whitelist; timer button now uses `:disabled="!canTimer"`; timer label shows "IndisponГӯvel neste estado" when blocked
- `ordens-servico/Detail.vue`: moved `await loadOS()` to `finally` block in `confirmTransition` вҖ” same root cause, same fix
- `store/auth.js`: replaced all `localStorage` calls with `sessionStorage`; each tab now holds an independent token so testing with multiple accounts in separate tabs works correctly; trade-off: session does not survive the tab being closed (acceptable for an internal-use app)

### [2026-05-16] Fix вҖ” `numero` column NOT NULL constraint crash on transfer creation

**Problem:**
Every attempt to create a transfer request returned a 500 Internal Server Error. The POST reached the backend and the INSERT was attempted, but MySQL rejected it immediately with:

```
IntegrityError: (1048, "Column 'numero' cannot be null")
```

**Root cause вҖ” chicken-and-egg ordering:**
The `numero` field is generated as `TRF-{year}-{id:04d}`, where `id` is the auto-increment primary key assigned by MySQL. To know the `id`, you need to INSERT the row first. But the column was defined as `NOT NULL`, so MySQL rejected the INSERT because `numero` was still `None` at that point.

The repository's `create()` method was:
```python
pt = PedidoTransferencia(**kwargs)   # numero = None
self.db.add(pt)
self.db.flush()                       # вҶҗ MySQL rejects: numero cannot be null
pt.numero = f"TRF-{year}-{pt.id:04d}"
```

**First solution (hotfix):**
Set a UUID-based placeholder before the flush to satisfy the constraint, then overwrite it with the real number after getting the id:

```python
pt.numero = f"TEMP-{uuid.uuid4().hex}"  # satisfies NOT NULL + UNIQUE
self.db.add(pt)
self.db.flush()                           # gets id
pt.numero = f"TRF-{year}-{pt.id:04d}"   # overwrites with real numero
```

**Why it was rejected:**
Although functional, this is a workaround that masks the actual design flaw. The column is semantically nullable during the brief window between INSERT and the UPDATE вҖ” the NOT NULL constraint was incorrectly applied. UUID strings as placeholder values are semantically meaningless and leave the door open to corruption if something fails between the two writes.

**Final solution (proper fix):**
Made `numero` nullable at both the ORM and DB level, which accurately reflects that the value genuinely does not exist until the row has an id:

- `models/transferencia.py`: changed `Mapped[str]` вҶ’ `Mapped[Optional[str]]` with `nullable=True`
- `repositories/transferencia_repository.py`: removed UUID placeholder; clean two-step flow (flush вҶ’ set numero)
- `alembic/versions/a1c3e5f7b9d2_...py`: migration to `ALTER TABLE pedidos_transferencia MODIFY COLUMN numero VARCHAR(50) NULL`

The column is still `unique=True`, so no two rows can ever share the same generated number. The `NULL` state is only possible inside the same uncommitted transaction.

### [2026-05-27] OS Detail вҖ” timer UTC offset bug + parts dropdown scroll

**Prompt:**
Two bugs in the OS detail page: (1) when concluding a diagnosis the timer jumps instantly to ~59 minutes; after concluding the OS it resets to the correct time. (2) The parts search dropdown can't be scrolled on mobile вҖ” the screen is too small to see all results.

**What was done:**

- **Timer UTC offset fix** (`frontend/src/views/oficina/Detail.vue`, `minutosTrabalhadosLive`): the backend returns `inicio_tempo_atual` as a naive datetime string (no timezone suffix, e.g. `"2026-05-27T14:00:00"`). JavaScript's `new Date()` was parsing this as *local time* (Portugal = UTC+1), while `Date.now()` is always UTC вҖ” producing a ~60-minute phantom elapsed time. Fix: append `'Z'` to the string before parsing if it has no timezone info, so it is correctly treated as UTC. Guard checks for existing `'Z'` or `'+'` to avoid double-applying.

- **Parts dropdown scroll** (`frontend/src/views/oficina/Detail.vue`, `.peca-dropdown`): changed `overflow: hidden` to `overflow-y: auto`, added `max-height: 260px` and `-webkit-overflow-scrolling: touch`. The dropdown now scrolls independently within a fixed height, so all results are reachable on small screens without scrolling the whole page.
